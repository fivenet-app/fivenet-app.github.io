FiveNet verwendet **CalVer** als Versionsschema. Das bedeutet, dass die Versionsnummer auf dem Veröffentlichungsdatum basiert und nicht auf einer klassischen semantischen Version wie `1.2.3`.

In der Regel sehen Sie eine dieser beiden Formen:

- Release-Version: `v2026.3.1`
- Entwicklungs-Version: `v2026.3.1-55-g07bf5c8e5`

## Release-Version

Eine vollständige Release-Version sieht so aus:

```text
v2026.3.1
```

Dieses Format ist:

```text
vYYYY.M.P
```

- `YYYY` = Veröffentlichungsjahr
- `M` = Veröffentlichungsmonat
- `P` = Patch- oder Revisionsnummer für diesen Monat

Im Beispiel `v2026.3.1` bedeutet das:

- `2026` steht für das Jahr 2026
- `3` steht für März
- `1` steht für das erste veröffentlichte Release in diesem Monat

## Entwicklungs-Version

Eine Entwicklungs-Version sieht so aus:

```text
v2026.3.1-55-g07bf5c8e5
```

Sie basiert auf derselben Release-Version, enthält aber zusätzlich angehängte Git-Metadaten:

```text
vYYYY.M.P-COMMITS-gHASH
```

- `v2026.3.1` = das nächstgelegene Release-Tag, auf dem der Build basiert
- `55` = der Build enthält 55 Commits nach diesem Tag
- `g07bf5c8e5` = der kurze Git-Commit-Hash der exakten Quellcode-Revision

Praktisch bedeutet das, dass der Build aus Code erstellt wurde, der neuer ist als das Release `v2026.3.1`, aber noch nicht als nächstes vollständiges Release veröffentlicht wurde.

## Bedeutet "Dev" automatisch instabil?

Nicht unbedingt.

Eine Entwicklungs-Version bedeutet lediglich, dass der Build aus einem Commit zwischen zwei getaggten Releases erstellt wurde. Das bedeutet **nicht** automatisch, dass der Build kaputt, unsicher oder für den Produktionseinsatz ungeeignet ist.

Der Hauptunterschied liegt in der Versionskennzeichnung:

- Eine Release-Version ist ein getaggtes, vollständiges Release.
- Eine Entwicklungs-Version ist ein Build von einem bestimmten Git-Commit nach dem letzten getaggten Release.

Da der Commit-Hash enthalten ist, lassen sich Entwicklungs-Versionen oft sogar leichter auf den exakten Quellcode zurückführen.

## Kurze Beispiele

- `v2026.3.1`
  Das erste getaggte Release aus dem März 2026.
- `v2026.3.1-55-g07bf5c8e5`
  Ein Build 55 Commits nach `v2026.3.1`, basierend auf dem Commit `07bf5c8e5`.

## Zusammenfassung

Wenn Sie sich nur eine Sache merken, dann diese:

::callout{icon="i-mdi-information-outline"}
`v2026.3.1` ist ein getaggtes Release, während `v2026.3.1-55-g07bf5c8e5` ein neuerer Build von einem bestimmten Git-Commit nach diesem Release ist. Eine Entwicklungs-Version ist nicht automatisch instabil.
::

## So prüfen Sie Ihre Version

::code-group

```bash [Docker / Docker Compose]
# Setzt voraus, dass Sie sich im Verzeichnis mit Ihrer docker-compose.yml / docker-compose.yaml befinden
docker compose exec fivenet fivenet --version
```

```bash [Kubernetes]
# Setzt voraus, dass FiveNet mit Helm bereitgestellt wurde und kubectl mit Ihrem Cluster sowie dem FiveNet-App-Namespace verbunden ist
kubectl exec deployment/fivenet -- fivenet --version
# Pruefen Sie zusaetzlich den in Ihren Helm-Values verwendeten image.tag, um zu sehen, welche Image-Version aktuell genutzt wird
```

```bash [Binary (Linux)]
# Setzt voraus, dass sich das fivenet-Binary in Ihrem PATH befindet
fivenet --version
```

```bash [Binary (Windows)]
# Setzt voraus, dass sich die fivenet.exe in Ihrem PATH befindet
fivenet.exe --version
```

::
